> На відміну від SQL де схема фіксована, в MongoDB ти сам вирішуєш як організувати дані. Ця свобода потребує свідомих рішень — неправильна схема на великих даних виправляється дуже болісно.

---

## Embedding vs Referencing

Головне рішення при проектуванні — зберігати пов'язані дані **в одному документі** чи в **окремих колекціях з посиланням**.

### Коли Embedding

```js
// все в одному документі
{
  _id: ObjectId("..."),
  title: "MongoDB Course",
  instructor: "Alice",
  lessons: [
    { title: "Intro", slides: "..." },
    { title: "CRUD", slides: "..." }
  ]
}
```

Embed коли:
- Дані зазвичай читаються разом
- Відношення "has-a" / "contains" (курс містить уроки)
- Дані оновлюються разом
- Дані архівуються разом
- Простіша модель важливіша за гнучкість

### Коли Referencing

```js
// courses колекція
{ _id: ObjectId("c1"), title: "MongoDB Course" }

// lessons колекція — окремо
{ _id: ObjectId("l1"), courseId: ObjectId("c1"), title: "Intro" }
{ _id: ObjectId("l2"), courseId: ObjectId("c1"), title: "CRUD" }
```

Reference коли:
- Висока кардинальність (один курс → тисячі уроків)
- Дані ростуть без меж
- Занадто багато дублювання
- Дані записуються в різний час
- Дочірня сутність може існувати без батьківської
- Розмір документа підходить до ліміту 16MB

## Duplicate Data — коли дублювання виправдане

Денормалізація (дублювання даних між колекціями) — **нормальна практика** в MongoDB якщо це прискорює читання і дані не змінюються часто.

| Тип | Опис | Приклад |
|---|---|---|
| **Immutable** | Ніколи не змінюється — ідеально для дублювання | Дата створення акаунту |
| **Temporal** | Важливо зберегти історичне значення | Адреса доставки на момент замовлення |
| **Sensitive to staleness** | Потребує частих оновлень | Кількість товару на складі |
| **Not sensitive to staleness** | Може бути трохи застарілим | Список рекомендованих товарів |

**Логіка вибору:**
- Дані immutable або temporal → дублюй сміливо
- Дані часто оновлюються → краще reference + `$lookup`
- Дані можуть бути трохи застарілими → дублюй + фоновий job для оновлення

## Atomicity

Write операція в MongoDB **атомарна на рівні одного документа**. Навіть якщо оновлення зачіпає вкладені документи — або все оновлюється, або нічого.

```js
// атомарна операція — один документ з вкладеними даними
db.orders.updateOne(
  { _id: orderId },
  { $set: { status: "shipped" }, $push: { history: { event: "shipped", date: new Date() } } }
)
```

Якщо потрібна атомарність між **кількома документами** → multi-document транзакції:

```js
const session = client.startSession()
session.withTransaction(async () => {
  await orders.updateOne({ _id: orderId }, { $set: { status: "paid" } }, { session })
  await inventory.updateOne({ _id: itemId }, { $inc: { qty: -1 } }, { session })
})
```

⚠️ Транзакції дорогі. Якщо часто потрібні транзакції — це сигнал що схема спроектована неправильно. Хороша схема мінімізує потребу в транзакціях.

## Інші практичні поради

### Індексуй те що часто запитуєш

Поля в `find()` фільтрах, `sort()`, `$lookup`, `$group` — кандидати на індекс. Детальніше: [[MongoDb - Indexes]].

Кожен індекс: мінімум 8KB простору + уповільнює запис. Моніторь використання індексів в проді.

### TTL для тимчасових даних

```js
// автовидалення сесій через 30 хвилин
db.sessions.createIndex({ updatedAt: 1 }, { expireAfterSeconds: 1800 })
```

### Маленькі документи — оптимізація

Якщо колекція складається з дуже маленьких документів:
- Встановлюй `_id` явно замість автогенерованого ObjectId (економить місце)
- Скорочуй назви полів (`last_name` → `lname`) — економить байти на кожному документі
- Embed маленькі документи один в одний

### Hardware constraint

Великі документи → більше RAM → читання з диска → деградація. Проектуй схему так щоб запити повертали тільки потрібні поля.

## Ключова ідея

```
SQL підхід:    нормалізуй все → join при читанні
MongoDB підхід: денормалізуй якщо читаєш разом → embed
               нормалізуй якщо часто оновлюєш або висока кардинальність → reference
```
