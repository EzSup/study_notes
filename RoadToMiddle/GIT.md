# Merge, Rebase, Cherry-pick
## Merge
Об'єднує дві гілки, створюючи **новий "merge commit"**, який містить зміни обох гілок.
Коли юзати:
- Коли хочеш **об'єднати код із двох гілок, не змінюючи історію**.
- Це **безпечний** спосіб інтеграції змін.
До:
```
   A---B---C (main)
        \  
         D---E (feature-branch)
```
Після:
```
   A---B---C---M (main)
        \      /
         D---E (feature-branch)

```
## Rebase
Переносить коміти з однієї гілки на іншу **без створення merge-коміту**, змінюючи їхню базу.
Коли юзати:
- Щоб **очистити історію** перед злиттям у `main` (не створюючи зайвих merge-комітів).
- Якщо хочеш **уникнути "розгалужень" в історії** (`git log` буде чистішим).
До:
```
   A---B---C (main)
        \  
         D---E (feature-branch)
```

Після:
```
   A---B---C (main)
           \
            D'---E' (feature-branch)  
```
## Cherry-pick
Копіює конкретний коміт з однієї гілки в іншу **без зміни всієї історії**.
Коли використовувати:
- Якщо треба взяти **тільки один коміт** з іншої гілки без злиття всіх змін.
- Якщо виправлення знаходиться в іншій гілці, і його треба застосувати окремо.
До:
```
   A---B---C (main)
        \  
         D---E (feature-branch)
```
Після:
```
   A---B---C---E' (main)
        \  
         D---E (feature-branch)
```
# Робота з історією та скасування змін

## 1. Ролбек локальних змін (до коміту)

Повертає файли до стану останнього коміту, видаляючи все, що ти написав, але ще не закомітив. Коли юзати:

- Коли ти наговівся і хочеш почати з чистого листа. Команда: `git checkout -- .` або сучасна `git restore .`
    

---

## 2. Відміна локального коміту

Скасовує останній коміт, який ти зробив, але ще не відправив (push) в origin.

### Зі збереженням змін (Soft)

Коміт зникає, але всі твої напрацювання залишаються у "Staging area" (готові до нового коміту). Коли юзати:

- Забув додати файл або зробив помилку в тексті повідомлення. Команда: `git reset --soft HEAD~1`
    

### Без збереження змін (Hard)

Коміт і всі зміни в ньому видаляються безповоротно. Коли юзати:

- Якщо коміт був повністю помилковим і код у ньому не потрібен. Команда: `git reset --hard HEAD~1`
    

---

## 3. Видалити коміт в Origin

Видаляє коміт з історії на сервері. **Увага:** Це змінює історію, що може створити проблеми колегам. Коли юзати:

- Коли ти випадково заpush-ив секретні дані (паролі) або абсолютно непотрібне сміття. Команда: `git reset --hard HEAD~1` (локально) `git push origin <branch-name> --force` (силове оновлення сервера)
    

---

## 4. Ролбек коміту в Origin (Safe Revert)

Створює **новий коміт**, який робить зміни, протилежні вказаному коміту. Коли юзати:

- Це **найбезпечніший** спосіб на публічних гілках. Історія не видаляється, а просто "відкочується" вперед. Команда: `git revert <commit-hash>` `git push origin <branch-name>`
    

До: `A---B---C` Після: `A---B---C---D` (де D — це анти-C)

---

## 5. Ребейзнути origin на певний коміт

Примусове встановлення гілки в origin на конкретний стан. Коли юзати:

- Коли треба "відкотити" сервер на кілька кроків назад до стабільної версії. Команда: `git reset --hard <commit-hash>` `git push origin <branch-name> --force`
    

---

## 6. Список комітів у вигляді дерева

Дозволяє візуально зрозуміти, як розгалужувались і зливались гілки. Коли юзати:

- Завжди, коли хочеш зрозуміти структуру проєкту перед merge або rebase. Команда: `git log --graph --oneline --all`
    

> **Порада від Pragmatic Programmer:** Додай аліас для красивого дерева у свій `$PROFILE`, який ми щойно налаштували: `Set-Alias gl "git log --graph --oneline --all --decorate"` Тепер просто ввівши `gl`, ти побачиш всю карту проєкту.
